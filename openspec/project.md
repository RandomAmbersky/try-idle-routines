# Контекст проекта: try-idle-routines

Документ для OpenSpec, агентов и людей: что это за проект, какие инструменты используются и какие соглашения приняты. **Обновляйте этот файл** при смене версий Rust, целевых платформ или CI.

---

## Назначение и идея продукта

- **Рабочее название:** `try-idle-routines`.
- **Жанр:** **idle RTS** — стратегия в реальном времени с упором на **idle/incremental**-прогресс: базы, юниты, ресурсы и тик/пассивный рост там, где это уместно для жанра.
- **Реализация:** проект ведётся на **Rust** (игровая логика, симуляция, при необходимости CLI или бинарь под выбранную платформу).

---

## Текущее состояние репозитория

- **Стек выбран:** прикладной код — **Rust**; в корне есть **`Cargo.toml`** и бинарный крейт `try-idle-routines` (`src/main.rs`).
- Настроены **spec-driven**-процесс OpenSpec и интеграция с **Cursor**.
- Сборка Rust попадает в `target/`; в `.gitignore` указано `/target`.

---

## Технологии и инструменты

| Область | Сейчас |
|--------|--------|
| Спецификации и изменения | **OpenSpec**, схема **`spec-driven`** (`openspec/config.yaml`) |
| CLI | `openspec` (создание изменений, статус, инструкции по артефактам) |
| Редактор / агенты | **Cursor**: навыки в `.cursor/skills/` и команды в `.cursor/commands/` |
| Прикладной стек | **Rust** — idle RTS (логика и симуляция; рендер/UI — по мере появления в репо) |

### OpenSpec (spec-driven)

- Изменения живут под `openspec/changes/<имя-изменения>/`.
- Типичный набор артефактов для потока propose: **proposal** (что и зачем), **design** (как), **tasks** (шаги реализации); точный набор задаёт схема и `openspec status`.
- Имена изменений: **kebab-case** (например, `add-user-auth`).
- Перед реализацией: артефакты из `applyRequires` должны быть в состоянии «готово»; дальше — `/opsx:apply` (см. команды ниже).

### Cursor: команды и навыки

- **`/opsx-propose`** — новое изменение и генерация артефактов propose/design/tasks (по навыку `openspec-propose`).
- **`/opsx-apply`** — выполнение задач из изменения (`openspec-apply-change`).
- **`/opsx-archive`** — архивация завершённого изменения (`openspec-archive-change`).
- **`/opsx-explore`** — режим исследования требований (`openspec-explore`).

## Соглашения по разработке

1. **Именование OpenSpec-изменений:** kebab-case, осмысленное имя фичи или задачи.
2. **Источник правды по изменению:** файлы в `openspec/changes/<name>/` и вывод `openspec status` / `openspec instructions`.
3. **Контекст для ИИ:** этот файл (`openspec/project.md`) и при необходимости поле `context` в `openspec/config.yaml` — не дублировать огромные блоки в каждом артефакте; держать факты о проекте здесь.
4. **Git (локально):** для единообразия коммитов в этом клоне можно использовать скрипт `git-set-local.sh` (локальные `user.name` / `user.email` / при необходимости SSH). Не коммитьте секреты и не полагайте, что настройки глобальны для всех машин.
5. **Rust-проект:** основной бинарь — пакет `try-idle-routines` в корне; сборка и запуск: `cargo build`, `cargo run`. Дописать по мере необходимости MSRV или версию toolchain, `rustfmt`/`clippy`, стиль коммитов и правила PR.
